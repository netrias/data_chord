Pass-through filter controls — comparison

Stage 2's existing chips filter by mapping source (AI / Override / None). A pass-through filter would be a different axis — CDE type. Each variant below sketches a way to add it. Click controls to see filtering behavior on the same mock list.

The decision: add the control on the same axis (cheap, mixes axes) or separate it (clean, more UI weight). Goal: pick the option that makes the common review action easier without making the bar feel busy.
Baseline

Current chips (no pass-through filter)

What ships today. Source-only filtering. Pass-throughs sit mixed into All / AI / Override.

Mixes axes

A — Fifth chip in the same row

Drop "Pass-through" alongside the existing chips. Cheapest possible change. Counts no longer sum to total because pass-through cuts across the source categories.

Pros

  • Discoverable — visually parallel to existing filters
  • Zero net new UI elements; no new mental model to learn
  • One click reaches the answer ("just show me pass-throughs")

Cons

  • Mixes orthogonal axes — chips no longer mean the same thing
  • Counts (4 + 2 + 2 = 8) no longer match the filter set
  • Can't compose ("AI rec + pass-through" is not expressible)
Two axes, two rows

B — Second chip row for type

Source row stays clean. A second row introduces the type axis explicitly. Filters compose: select "AI Rec" + "Pass-through" to see exactly that intersection.

Type

Pros

  • Clean separation — each row is a single, coherent axis
  • Filters compose naturally; counts always sum within a row
  • Scales if a third axis (e.g., conformance) ever needs a filter

Cons

  • Doubles the vertical footprint above the list
  • "Type" chips may feel like overhead for a 1-of-2 distinction
  • Two control rows = more affordances to scan before reading list
Lightweight

C — "Hide pass-throughs" toggle

Single boolean control inline with search. Solves the most common case ("skip the trivial ones, focus on real review work") without introducing a new filter axis. Doesn't let you isolate pass-throughs — only suppress them.

Pros

  • Smallest possible UI footprint — one control, one bit of state
  • Matches the actual review intent (skip trivial → focus attention)
  • Doesn't compete with the source chips for visual weight

Cons

  • Asymmetric — can hide pass-throughs but can't isolate them
  • Less discoverable than chips; lives next to search rather than filters
  • Adds a hidden mode — a row count of "5 of 8" needs explaining
Segmented

D — Inline type segmented control

A compact 3-way segmented control on the search row. Reads as a tighter version of B — type axis is explicit but takes one line, not two. Still composes with source chips.

Pros

  • Single row but two axes — keeps vertical footprint of baseline
  • Composes with source chips; both filters can be combined
  • Segmented control visually signals "pick exactly one"

Cons

  • Search row gets dense — input + segmented control fight for space
  • Two different control styles (chips vs. segmented) on adjacent rows
  • Wraps awkwardly on narrow viewports
Hidden complexity

E — "Filters" dropdown menu

One button opens a popover holding both axes (and any future ones). Top bar stays uncluttered; trade-off is that filter state is no longer visible at a glance.

Pros

  • Top bar stays minimal — search + one button
  • Scales gracefully to many filter axes without cluttering the page
  • Active filter count badge keeps state visible at a glance

Cons

  • Filters are hidden behind a click — less ambient awareness
  • Loses the "click a chip, see counts shift" feedback loop entirely
  • Heavier interaction model for a 4 + 2 axis problem
Recommendation framing

How to choose

  • If the goal is "let reviewers skip pass-throughs to focus on real work": C (toggle). Smallest change, addresses the use case directly.
  • If reviewers need to isolate pass-throughs (e.g., spot-check a batch): B or D. Both expose the type axis explicitly.
  • If you expect more filter axes in the future (conformance %, fit, etc.): E (dropdown). Only option that doesn't keep growing the chrome.
  • Avoid A. The mixed-axis chip is cheap to ship but breaks the implicit contract that source chip counts sum to the total — a small lie that compounds.
